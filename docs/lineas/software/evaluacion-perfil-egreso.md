# Evaluacion del perfil de egreso: linea de Software

## Proposito

La evaluacion del perfil de egreso en la linea de Software tiene como finalidad verificar que el estudiante demuestra, de manera integrada, las competencias de Ingenieria de Requerimientos, Ingenieria de la Informacion, Programacion y Calidad de Software mediante una solucion final de software, un portafolio sintetico de evidencias y una sustentacion tecnica profesional.

Esta evaluacion no busca revisar todos los entregables progresivos desarrollados durante la carrera. Su foco es comprobar que, al cierre del proceso formativo, el estudiante puede integrar lo aprendido en un producto funcional, documentado, defendible y alineado con una necesidad real.

La evaluacion corresponde a CE02, Ingenieria de Software: gestiona y desarrolla software de manera eficiente y efectiva, basandose en estandares internacionales de calidad para lograr el control y aseguramiento de la calidad segun el contexto de la organizacion.

## Principio de evaluacion

La evaluacion debe centrarse en evidencias finales e integradoras. Los artefactos progresivos son importantes para la formacion, pero no todos deben ser recalificados en el momento de egreso.

Por ello, el estudiante presenta:

- Un producto final de software funcional.
- Un dossier tecnico final.
- Evidencias de datos, calidad, despliegue y operacion.
- Una demostracion funcional.
- Una sustentacion tecnica.

Los artefactos historicos pueden usarse como soporte, pero la evaluacion principal debe recaer sobre la version final del sistema y sus evidencias consolidadas.

## Entregables obligatorios

### 1. Dossier final de requerimientos y diseno

Competencia asociada: CE021, Ingenieria de Requerimientos.

Alcance de la competencia: define, analiza y valida requerimientos funcionales y no funcionales, y disena la arquitectura y el comportamiento del sistema desde la perspectiva del usuario, del negocio y del contexto organizacional.

Debe incluir:

- Especificacion de requerimientos validada.
- Historias de usuario o casos de uso.
- Criterios de aceptacion.
- Prototipo navegable validado.
- Diseno arquitectonico del sistema.
- Diseno detallado del sistema.
- Evidencias de validacion con usuarios o stakeholders.

Evidencias evaluadas:

- CE0211: Especificacion de requerimientos validada.
- CE0212: Prototipo navegable validado.
- CE0213: Diseno arquitectonico del sistema.
- CE0214: Diseno detallado del sistema.

### 2. Plataforma de datos del sistema

Competencia asociada: CE022, Ingenieria de la Informacion.

Alcance de la competencia: modela, implementa y administra estructuras de datos, garantizando integridad, consistencia, rendimiento, seguridad y disponibilidad de la informacion.

Debe incluir:

- Modelo de datos relacional.
- Diccionario de datos.
- Base de datos implementada.
- Consultas SQL relevantes.
- Procedimientos almacenados, funciones o triggers cuando correspondan.
- Seguridad y administracion de datos.
- Evidencias de funcionamiento.
- Documentacion tecnica de administracion.

Evidencias evaluadas:

- CE0221: Modelo de datos relacional documentado y validado.
- CE0222: Base de datos relacional implementada y consultable.
- CE0223: Motor transaccional programado y optimizado.
- CE0224: Base de datos empresarial administrada, asegurada y operativa.

### 3. Proyecto integrador de software

Competencia asociada: CE023, Programacion.

Alcance de la competencia: desarrolla e integra soluciones de software aplicando modularidad, desacoplamiento, patrones de diseno y buenas practicas para lograr sistemas funcionales y mantenibles.

El estudiante debe presentar un proyecto de software funcional que resuelva un problema real y evidencie:

- Arquitectura adecuada al contexto.
- Persistencia de datos.
- Integracion de componentes.
- Interfaz de usuario.
- Funcionalidades operativas.
- Despliegue funcional o instalacion verificable.
- Calidad tecnica del codigo.

El proyecto puede implementarse como:

- Aplicacion desktop.
- Sistema web.
- Aplicacion empresarial full-stack.
- Sistema distribuido.
- Aplicacion movil.
- Solucion hibrida.

No se exige que el estudiante construya todas las modalidades. La evaluacion debe considerar la modalidad elegida y su pertinencia frente al problema abordado.

Evidencias posibles segun la naturaleza del proyecto:

- CE0231: Aplicacion desktop funcional con persistencia de datos.
- CE0232: Sistema de informacion web integrado con base de datos.
- CE0233: Aplicacion empresarial full-stack desplegable.
- CE0234: Sistema distribuido escalable basado en servicios interoperables.
- CE0235: Aplicacion movil integrada con servicios digitales.

Para la evaluacion de egreso, el jurado debe valorar la evidencia CE023 que corresponda al tipo de solucion presentada. Si el proyecto integra mas de una modalidad, puede evidenciar mas de una CE023.

### 4. Dossier final de calidad y operacion

Competencia asociada: CE024, Calidad de Software.

Alcance de la competencia: asegura la calidad del producto y del proceso mediante pruebas, CI/CD, metricas, revision tecnica, gestion de deuda tecnica, auditoria y mejora continua.

Debe incluir:

- Casos de prueba automatizados o pruebas sistematizadas.
- Reportes de ejecucion.
- Reporte de cobertura cuando aplique.
- Evidencias de integracion y despliegue.
- Gestion tecnica del proyecto.
- Registro de incidencias.
- Revisiones de codigo o evidencias de control tecnico.
- Metricas de calidad.
- Auditoria tecnica.
- Plan de mejora y evolucion.

Evidencias evaluadas:

- CE0241: Suite de pruebas automatizadas.
- CE0242: Pipeline de integracion y despliegue continuo.
- CE0243: Gestion tecnica y aseguramiento de calidad.
- CE0244: Auditoria tecnica y evolucion del sistema.

### 5. Presentacion y sustentacion final

La sustentacion final permite verificar que el estudiante no solo entrego artefactos, sino que comprende y defiende la solucion construida.

#### Video pitch

Duracion sugerida: 1 a 3 minutos.

Debe comunicar:

- Problema abordado.
- Solucion propuesta.
- Usuarios o beneficiarios.
- Funcionalidades principales.
- Valor agregado de la solucion.
- Resultados obtenidos.

#### Sustentacion tecnica

Debe incluir:

- Demo funcional del sistema.
- Explicacion tecnica de la solucion.
- Justificacion de decisiones de diseno y arquitectura.
- Evidencia de integracion de componentes.
- Evidencia de calidad y despliegue.
- Respuesta a preguntas del jurado.

La presentacion y sustentacion puede registrarse como una evidencia transversal de cierre del perfil de egreso. Si se desea codificarla, conviene definirla como evidencia integradora de la linea y no como parte de CE021, porque evalua el conjunto completo de la solucion.

## Rubrica de evaluacion del perfil de egreso

### 1. Ingenieria de Requerimientos: 25%

| Nivel | Descripcion |
| --- | --- |
| Excelente | Requerimientos, prototipo, arquitectura y diseno son completos, consistentes, validados y alineados con el problema. |
| Bueno | Presenta la mayoria de elementos con observaciones menores. |
| Basico | Presenta elementos incompletos o parcialmente alineados. |
| Deficiente | No evidencia coherencia entre problema, requerimientos, diseno y solucion. |

### 2. Ingenieria de la Informacion: 20%

| Nivel | Descripcion |
| --- | --- |
| Excelente | Modelo, implementacion, programacion y administracion de datos son consistentes, seguros y adecuados al sistema. |
| Bueno | Presenta funcionamiento adecuado con observaciones menores. |
| Basico | Presenta limitaciones de diseno, seguridad, administracion u optimizacion. |
| Deficiente | No demuestra una solucion de datos funcional. |

### 3. Programacion: 30%

| Nivel | Descripcion |
| --- | --- |
| Excelente | El proyecto es funcional, integra adecuadamente sus componentes y evidencia una arquitectura solida y mantenible. |
| Bueno | Cumple los requerimientos principales con observaciones menores. |
| Basico | Presenta funcionalidades incompletas o limitaciones importantes. |
| Deficiente | El proyecto no es funcional o no puede demostrarse. |

### 4. Calidad de Software: 15%

| Nivel | Descripcion |
| --- | --- |
| Excelente | Evidencia pruebas, despliegue, aseguramiento de calidad, auditoria tecnica y mejora continua. |
| Bueno | Presenta mecanismos basicos de calidad y despliegue. |
| Basico | Presenta evidencias parciales de calidad. |
| Deficiente | No evidencia practicas de calidad. |

### 5. Presentacion y sustentacion tecnica: 10%

| Aspecto | Peso interno |
| --- | --- |
| Video pitch | 20% |
| Demo funcional | 30% |
| Explicacion tecnica | 25% |
| Justificacion de decisiones | 15% |
| Respuesta a preguntas | 10% |

## Escala de calificacion

| Nivel | Rango |
| --- | --- |
| Excelente | 90 a 100 |
| Bueno | 80 a 89 |
| Basico | 70 a 79 |
| Deficiente | Menor a 70 |

## Criterio de cierre

Para considerar lograda la linea de Software en el perfil de egreso, el estudiante debe demostrar una solucion final funcional y defendible. No basta con presentar documentos aislados; debe existir coherencia entre problema, requerimientos, datos, arquitectura, implementacion, calidad, despliegue y sustentacion.

La evidencia final debe responder una pregunta central: el estudiante puede construir, explicar, probar, desplegar y mejorar una solucion de software con criterio profesional?
